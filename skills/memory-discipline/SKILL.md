# Memory Discipline

Правила ведения постоянной файловой памяти проекта (`~/.claude/projects/<project>/memory/`). Цель — после перерыва или перезагрузки восстанавливать контекст проекта как гроссмейстер позицию: распознаванием паттернов, а не реконструкцией с нуля.

## 1. Ритуал ориентировки (начало сессии / после перерыва)

Прежде чем действовать, пройди четыре стадии:

1. **Ориентировка** — прочитай `MEMORY.md` (индекс памяти проекта); он уже в контексте — освежи, что там есть, и открой релевантные файлы памяти целиком.
2. **Обследование** — `git log --oneline -10` и `git status`; сверь с последним известным состоянием из памяти. Ищи изменения, сделанные пользователем между сессиями (правки документов, переименования).
3. **Проработка** — если память и репозиторий расходятся, сначала выясни, что поменялось и почему (diff, чтение файлов), обнови устаревшие файлы памяти.
4. **Доказательство** — только после этого принимай решения и правь файлы. Не действуй уверенно по устаревшей картине.

## 2. Чанкинг: что и как записывать

- **Один файл = один осмысленный факт** с типом `user` / `feedback` / `project` / `reference`, полями `**Why:**` и `**How to apply:**` (для feedback/project) и ссылками `[[имя-другой-памяти]]`.
- **Записывать только невыводимое.** Код, коммиты, документы — уже «на доске»: их не дублировать. В память идёт то, чего нет ни в одном diff: почему выбрали технологию, что пользователь поправил и что это означает, конвенции именования, привычки пользователя.
- **Сырые транскрипты и логи не хранить** — это «случайная расстановка фигур», она не восстанавливается. Хранить дистиллят: решение + причина + как применять.
- **Относительные даты → абсолютные** («вчера» → «02.07.2026»).

## 3. Обновление на вехах, а не «в конце»

Записывай память сразу, когда происходит одно из:
- пользователь принял решение или отменил прежнее;
- пользователь дал фидбек о том, как работать (даже неявный — например, сам поправил твой документ);
- завершён этап работы, изменилась стадия проекта;
- выяснился факт о пользователе или внешней системе, который понадобится через недели.

«Конца сессии» может не быть — контекст сжимается внезапно.

## 4. Гигиена банка паттернов

- Перед созданием файла проверь индекс: если факт уже покрыт — обнови существующий файл, не плоди дубликаты.
- Устаревшую или опровергнутую память удаляй или исправляй немедленно — неверный паттерн хуже отсутствующего.
- Каждый новый файл — одна строка в `MEMORY.md`: `- [Заголовок](файл.md) — краткий хук`. Содержимое в индекс не выносить.
- Периодически (или по команде) выполняй консолидацию: слить дубли, починить ссылки, убрать мёртвые строки индекса.

## 5. Проверка перед ответом после перерыва

Если сессия возобновилась после перерыва и ты собираешься сослаться на факт из памяти (файл, флаг, оценку, имя) — сначала проверь, что он ещё существует и не изменён пользователем. Память отражает момент записи, а не настоящее.
